# Wiseman Auto System - Product Requirements Document

## 1. 概要

### プロダクトビジョン
介護ソフト「ワイズマンシステムSP」のデータを、Python RPA（pywinauto）によるGUI自動操作でGCPクラウドに連携する。物理USBドングルの制約を突破し、介護現場のデータをクラウドに解放する。

### 背景と課題
- ワイズマンシステムSPは**ASP型サービスだが、クライアントは.NETネイティブWindowsアプリ**（MDI構成）
  - サーバー: ワイズマン社がクラウドでホスティング（ASP）
  - クライアント: 各PCに.NETアプリをインストール（ブラウザは使わない）
  - ランタイム: .NET Framework 3.5
- 各PCにUSBキー（ドングル）またはライセンスID認証が必要
- データはワイズマンASPサーバー内に閉じ込められ、外部クラウドとの連携手段がない
- 手作業によるデータ転記が発生し、介護記録作成に時間がかかる
- 遠隔からのデータ確認・分析・加工が不可能

### 解決アプローチ
Python RPA（pywinauto）がワイズマンの.NET WinForms GUIを自動操作し、データ抽出→GCPクラウド転送→分析の流路を構築する。Pythonデスクトップアプリ（Hub）がRPA操作・クラウド同期・スケジューリングを統括する。開発はTeamViewerによる遠隔制御で完結させ、現場訪問を最小化する。

## 4. フェーズ計画

### Phase 1: PoC（概念実証） - 24時間（実働）

**ゴール**: ログイン→データ抽出→GCS転送の1本パイプラインを完走

| 機能 | 詳細 | 優先度 |
|------|------|--------|
| 自動ログイン | pywinautoでワイズマン起動・ログイン画面操作 | P0 |
| 画面遷移 | MDIメニューから帳票画面（ケア記録集計表等）に遷移 | P0 |
| データ抽出 | CSV出力ボタン→保存ダイアログ→CSV保存 or グリッド直接読取 | P0 |
| GCS転送 | CSVファイルをCloud Storageへ自動アップロード | P0 |
| エラー処理 | リトライ・ログ・スクリーンショット保存（基本レベル） | P1 |

**成功基準**:
1. pywinautoによるWiseman起動・ログイン完遂
2. ケア記録集計表等の帳票からCSVが出力される
3. CSVファイルがGCS上に到達
4. 上記を人手介入なしで完走

**スコープ外**: exe化、自動更新、スケジューラ、BigQuery、データ入力

---

### Phase 2: MVP（最小実用製品） - 2-4週間

| 機能 | 詳細 |
|------|------|
| exe化 | PyInstallerでシングルファイルexe。Python不要で動作 |
| スケジューラ | APSchedulerによる定期実行（日次/時次） |
| システムトレイ | pystrayによる状態表示（稼働中/アイドル/エラー） |
| BigQuery連携 | Cloud FunctionsでCSV→BigQuery自動ロード |
| 複数帳票対応 | TOML設定で抽出対象帳票・セレクタを指定 |
| グリッド直接読取 | CSV出力できないデータのグリッドコントロール読取 |
| CI/CD | GitHub Actions → Windows exe ビルド → GCS配置 |
| エラー通知 | スクリーンショット + エラーログのクラウド送信 |

## 6. 技術スタック

### クライアント側

| レイヤー | 技術 | 理由 |
|---------|------|------|
| **RPA（主）** | **pywinauto** (UIA backend) | Windows標準コントロールに最適。MDI対応、アクセシビリティセレクタ |
| **RPA（副）** | **PyAutoGUI** | カスタム描画コントロールのフォールバック（画像マッチング/座標） |
| スケジューリング | APScheduler | 軽量、インプロセス |
| デスクトップUI | pystray + tkinter | 最小構成のシステムトレイ |
| GCP SDK | google-cloud-storage, google-cloud-pubsub | 公式クライアント |
| パッケージング | PyInstaller → Nuitka | 段階移行 |
| 設定 | TOML | Python 3.11+ stdlib対応。セレクタ情報も外部化 |
| 認証 | USBドングルのみ | アプリ内ログイン画面なし（ADR-007） |

### GCP側（月額推定: $0.25-2.00）

| サービス | 用途 | 推定月額 |
|---------|------|---------|
| Cloud Storage (Standard) | データレイク | ~$0.25 |
| BigQuery | 分析ウェアハウス | $0 (Free Tier) |
| Pub/Sub | 指示キュー | $0 (Free Tier) |
| Cloud Functions 2nd gen | ETL/API | $0 (Free Tier) |

## 7. 開発・デプロイフロー

```
Developer (Mac)
    ├── Claude Code で開発
    │   - RPA以外のコード（クラウド、設定、更新）をmacOSで開発・テスト
    │   - RPAコードはWindows専用 → モック実装でユニットテスト
    │
    ├── TeamViewer → Client PC で実機テスト
    │   - USBドングル + 本番Wiseman でE2E検証
    │   - pywinautoのセレクタ調査・確定
    │   - Inspect.exe / Spy++ でコントロール構造を調査
    │
    └── git push → GitHub
            ├── CI: lint + test (ubuntu)
            ├── Build: PyInstaller exe (windows-latest)
            └── Release: GCS upload + manifest更新
                    │
                    ▼
            Client PC: 自動更新で取得
```

## 8. セキュリティ要件

| カテゴリ | 要件 |
|---------|------|
| データ保護 | 介護データ（個人情報）→ APPI準拠確認。東京リージョン必須 |
| 通信暗号化 | GCPとの通信は全てHTTPS (TLS 1.2+) |
| 認証情報 | Wiseman: USBドングルのみ（ADR-007）。GCPキー: 最小権限のサービスアカウント |
| ドングル | 仮想化禁止。不在検出→クラウド通知→手動対応 |
| exe配布 | コード署名証明書でDefender誤検知防止 |
| 監査ログ | 全データ操作のログ記録 |

## 9. リスクと対策

| リスク | 確率 | 影響 | 対策 |
|-------|------|------|------|
| WisemanのUI変更でRPA破損 | 中 | 高 | アクセシビリティセレクタ + TOML外部化。複数セレクタ戦略でフォールバック |
| MDI子ウィンドウの特定困難 | 中 | 中 | タイトルバーパターンマッチ + コントロールツリー探索 |
| TeamViewerレイテンシでRPAタイミング問題 | 高 | 中 | `wait("visible")`/`wait("ready")`活用。十分なタイムアウト |
| データグリッドがカスタム描画 | 中 | 高 | pywinauto UIA → Win32 → PyAutoGUI画像マッチングの3段階フォールバック |
| CSV文字エンコーディング（Shift-JIS） | 中 | 中 | エンコーディング自動検出 + 設定で指定可能 |
| USBドングル未認識 | 低 | 高 | エラーダイアログ検出→クラウド通知 |
| exeマルウェア判定 | 高 | 中 | コード署名証明書 |
| macOSでRPAテスト不可 | 確実 | 中 | モック実装 + TeamViewer実機テストのワークフロー確立 |
| APPI違反 | 低 | 致命 | 法務確認、仮名化、データ保持ポリシー |
